# Review Change Verdict

## 최종 답변 우선 규칙

이 문서 아래에 고정 라벨, 표, JSON, 예시가 있어도 사용자에게 보이는 최종 답변은 이 규칙을 먼저 따른다.

- 결론, 판정, 변경사항을 첫 문장에 둔다.
- 기본 답변은 쉬운 한국어 2~4문장으로 끝낸다.
- 세부 근거, 명령, 파일 목록은 결과를 이해하거나 검증하는 데 꼭 필요할 때만 넣는다.
- 필수 출력 형식, JSON 계약, 사용자 요청 형식은 보존한다.
- 내부 정리용 라벨과 템플릿 문구는 그대로 노출하지 않는다.
- 불확실하면 추측하지 말고 확인해야 할 사실만 짧게 말한다.
- 사용자가 묻지 않은 다음 작업 추천은 넣지 않는다.

## 원자성 원칙

- 이 스킬은 이 `SKILL.md`와 같은 폴더의 `references/`, `scripts/`, `assets/`만으로 이해되고 실행되어야 한다.
- sibling skill, 전역 context 파일, 외부 설치 스킬에 의존하지 않는다.
- 한 번에 이 스킬이 맡은 일 하나만 수행한다. 범위를 벗어나면 다른 스킬 이름을 짧게 말하고 중지한다.

## 품질 기준

- 출력량보다 명확성, 깊이, 건전한 판단, 오래가는 실질 기여를 봅니다.

## 목적

변경된 코드가 의도한 일을 안전하게 하는지 리뷰하고 `integrate` 또는 `hold` 판단을 냅니다. 이 스킬은 리뷰 전용이며 코드를 패치하지 않습니다.

## 경계

- `review-quality-check`: 고정 품질 체크리스트만 필요할 때 사용합니다.
- `review-security-check`: 보안만 볼 때 사용합니다.
- `review-failure-paths`: 실패 경로만 볼 때 사용합니다.
- `qa-change-scenarios`: 실행 시나리오 검증이 핵심일 때 사용합니다.

## 입력

- 변경 범위 또는 diff
- 변경 의도
- 관련 테스트 결과, 계획, 이슈, 요구사항

## 절차

1. 변경 의도와 범위를 고정합니다.
2. 변경된 코드가 의도와 맞는지 확인합니다.
3. 실제 결함을 file/line 근거로 우선순위화합니다.
4. 테스트 공백은 결함과 분리합니다.
5. 취향이나 선호는 blocker로 만들지 않습니다.
6. 최종 판단을 `integrate` 또는 `hold`로 냅니다.

## 판단 기준

- 출력량보다 명확성, 깊이, 건전한 판단, 오래가는 실질 기여를 봅니다.
- 긴 findings 목록보다 통합을 막는 실제 결함, 테스트 공백, 유지보수 위험을 정확히 분리하는 것이 중요합니다.
- 큰 diff 자체를 나쁘게 보지 말고, diff가 올바른 계약을 더 명확하고 검증 가능하게 만드는지 판단합니다.
- 근거가 약하면 `hold`로 꾸미지 말고 남은 확인 방법을 말합니다.

## 출력

```text
Verdict: integrate / hold
Findings:
Testing gaps:
Evidence:
Residual risk:
```

## 중지 조건

- 리뷰 범위가 모호합니다.
- 변경 의도가 없습니다.
- 근거 없이 결함이나 통합 가능 여부를 만들어야 합니다.
