# QA Bug Fix Proof

## 최종 답변 우선 규칙

이 문서 아래에 고정 라벨, 표, JSON, 예시가 있어도 사용자에게 보이는 최종 답변은 이 규칙을 먼저 따른다.

- 결론, 판정, 변경사항을 첫 문장에 둔다.
- 기본 답변은 쉬운 한국어 2~4문장으로 끝낸다.
- 세부 근거, 명령, 파일 목록은 결과를 이해하거나 검증하는 데 꼭 필요할 때만 넣는다.
- 필수 출력 형식, JSON 계약, 사용자 요청 형식은 보존한다.
- 내부 정리용 라벨과 템플릿 문구는 그대로 노출하지 않는다.
- 불확실하면 추측하지 말고 확인해야 할 사실만 짧게 말한다.
- 사용자가 묻지 않은 다음 작업 추천은 넣지 않는다.

## 원자성 원칙

- 이 스킬은 이 `SKILL.md`와 같은 폴더의 `references/`, `scripts/`, `assets/`만으로 이해되고 실행되어야 한다.
- sibling skill, 전역 context 파일, 외부 설치 스킬에 의존하지 않는다.
- 한 번에 이 스킬이 맡은 일 하나만 수행한다. 범위를 벗어나면 다른 스킬 이름을 짧게 말하고 중지한다.

## 품질 기준

- 출력량보다 명확성, 깊이, 건전한 판단, 오래가는 실질 기여를 봅니다.

## 목적

선택된 버그 수정이 원래 실패를 실제로 고쳤는지 확인합니다. 이 스킬은 검증 전용이며 수정 설계를 다시 하지 않습니다.

## 경계

- `analyze-failure-reproduction`: 원래 재현 절차가 아직 불명확할 때 사용합니다.
- `analyze-root-cause`: 수정 전에 원인을 찾아야 할 때 사용합니다.
- `qa-change-scenarios`: 여러 변경 동작을 시나리오로 넓게 검증할 때 사용합니다.
- `review-change-verdict`: 변경 전체의 결함과 통합 여부를 리뷰할 때 사용합니다.

## 입력

- 원래 실패 또는 재현 절차
- 선택된 수정 후보 또는 변경 diff
- 관련 테스트, 명령, 로그, 수동 확인 경로

## 절차

1. 원래 실패와 수정 후보를 읽습니다.
2. 원래 재현을 가장 먼저 다시 실행합니다.
3. 재현 실행이 불가능하면 왜 불가능한지와 다음 증거를 적습니다.
4. 원래 실패 경로가 수정됐는지 확인합니다.
5. 회귀 보호가 실제인지 가장 작은 테스트나 명령으로 확인합니다.
6. 결과를 `fixed`, `not-fixed`, `inconclusive` 중 하나로 냅니다.

## 판단 기준

- 출력량보다 명확성, 깊이, 건전한 판단, 오래가는 실질 기여를 봅니다.
- 많은 보조 테스트보다 원래 재현을 없애는 증거와 같은 버그 재발을 막는 가장 작은 guard가 중요합니다.
- 수정이 우연히 통과한 것인지 근본 경로를 막은 것인지 구분하지 못하면 `inconclusive`로 둡니다.

## 출력

```text
Status: fixed / not-fixed / inconclusive
Original repro:
Fix checked:
Evidence:
Regression guard:
Remaining risk:
Next proof:
```

## 중지 조건

- 원래 실패가 다시 확인할 만큼 명확하지 않습니다.
- 수정 후보가 다른 변경과 섞여 분리할 수 없습니다.
- 증거 없이 성공을 주장해야 하는 상태입니다.
