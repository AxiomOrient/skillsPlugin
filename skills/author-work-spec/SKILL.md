# Author Work Spec

## 최종 답변 우선 규칙

이 문서 아래에 고정 라벨, 표, JSON, 예시가 있어도 사용자에게 보이는 최종 답변은 이 규칙을 먼저 따른다.

- 결론, 판정, 변경사항을 첫 문장에 둔다.
- 기본 답변은 쉬운 한국어 2~4문장으로 끝낸다.
- 세부 근거, 명령, 파일 목록은 결과를 이해하거나 검증하는 데 꼭 필요할 때만 넣는다.
- 필수 출력 형식, JSON 계약, 사용자 요청 형식은 보존한다.
- 내부 정리용 라벨과 템플릿 문구는 그대로 노출하지 않는다.
- 불확실하면 추측하지 말고 확인해야 할 사실만 짧게 말한다.
- 사용자가 묻지 않은 다음 작업 추천은 넣지 않는다.

## 원자성 원칙

- 이 스킬은 이 `SKILL.md`와 같은 폴더의 `references/`, `scripts/`, `assets/`만으로 이해되고 실행되어야 한다.
- sibling skill, 전역 context 파일, 외부 설치 스킬에 의존하지 않는다.
- 한 번에 이 스킬이 맡은 일 하나만 수행한다. 범위를 벗어나면 다른 스킬 이름을 짧게 말하고 중지한다.

## 품질 기준

- 출력량보다 명확성, 깊이, 건전한 판단, 오래가는 실질 기여를 봅니다.

## 목적

하나의 구현-ready 작업 명세를 작성하거나 갱신합니다. 이 스킬은 문서 작성 전용이며 코드를 패치하지 않습니다.

## 경계

- `plan-implementation-scope`: 대화 안에서 짧은 구현 계획만 필요할 때 사용합니다.
- `execute-product-completion`: 명세를 바탕으로 실제 구현까지 진행할 때 사용합니다.
- `execute-docs-renewal`: 이미 있는 README나 운영 문서를 코드에 맞게 갱신할 때 사용합니다.

## 입력

- 요청, 계획, 이슈, PRD, 회의 메모, 기존 spec
- 저장소 근거가 필요한 경우 관련 파일, 설정, 테스트, 명령

## 절차

1. 요청과 근거를 읽습니다.
2. 요구사항, 비목표, 제약, 열린 질문을 분리합니다.
3. 구현 범위와 변경하지 않을 범위를 명확히 씁니다.
4. 완료 기준과 검증 명령을 적습니다.
5. 구현자가 바로 사용할 수 있는 하나의 work spec만 작성합니다.

## 출력

```text
Work spec:
- Goal:
- Scope in:
- Scope out:
- Requirements:
- Interfaces or files:
- Acceptance checks:
- Verification:
- Open questions:
```

## 중지 조건

- 요구사항이 명세로 고정할 만큼 분명하지 않습니다.
- 핵심 제약을 뒷받침할 현재 근거가 없습니다.
- 하나의 bounded spec이 아니라 큰 혼합 문서가 됩니다.
